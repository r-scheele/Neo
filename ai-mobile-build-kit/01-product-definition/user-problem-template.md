# User Problem Template

Use this to avoid building a technically interesting app that does not solve a painful problem.

## Target User

- User segment: `[WHO]`
- Context: `[WHEN_AND_WHERE_THE_PROBLEM_HAPPENS]`
- Skill level: `[BEGINNER / INTERMEDIATE / ADVANCED / MIXED]`
- Device habits: `[PHONE_CONTEXT, FREQUENCY, NOTIFICATIONS, OFFLINE_NEEDS]`

## Main Pain

Write the pain in the user's language:

`I struggle with [PROBLEM] because [CAUSE], which leads to [CONSEQUENCE].`

Example:

`I struggle with remembering my daily habits because my schedule changes, which leads to losing momentum after a few days.`

## Current Workarounds

- Workaround 1: `[SPREADSHEET / NOTES / EXISTING_APP / MANUAL_PROCESS]`
- What works about it: `[BENEFIT]`
- What fails about it: `[LIMITATION]`

## Desired Outcome

The user wants:

- Functional outcome: `[WHAT_GETS_DONE]`
- Emotional outcome: `[HOW_THEY_WANT_TO_FEEL]`
- Social outcome if relevant: `[HOW_IT_AFFECTS_OTHERS]`

## Problem Frequency

- How often does this problem happen?
- What triggers it?
- What happens if the user ignores it?
- Is the pain strong enough for a dedicated app?

## Edge Cases

- New user with no data.
- Returning user with old data.
- User with very long content.
- User offline.
- User denies permissions.
- User starts but does not finish setup.

## Done Looks Like

This problem is clear when the MVP feature list directly maps to the pain, not to generic app conventions.
